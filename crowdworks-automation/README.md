# クラウドワークス案件応募 半自動化ツール

新着案件を定期的にチェックし、AIが応募文のドラフトを作成してレポートにまとめるツールです。
**送信は自動化しません**（講師の助言どおり半自動）。レポートを見て内容を確認・修正してから、自分でクラウドワークス上で応募してください。

このツールは特定のエディタ/AIツール(Cursor、Claude Code等)に依存しません。
**通常のmacOSターミナル(Terminal.app等)だけで動作します。**

## 必要環境

| 項目 | 必要バージョン | 確認コマンド |
|---|---|---|
| OS | macOS | - |
| Node.js | v18以上(推奨: 最新LTS) | `node --version` |
| pnpm | v8以上 | `pnpm --version` |

Node.js/pnpmが入っていれば、Cursor・Claude Code・VS Code・素のTerminal.appなど、
どのターミナルからでも同じように動作します(このツール自体は特定のツールの環境変数やPATHには一切依存しません)。

## インストール(初回・環境構築)

Node.js/pnpmが未導入の場合は、**通常のmacOSターミナル(Terminal.app)**で以下を実行してください。

```bash
# 1. Homebrewが無ければインストール(すでにある場合は不要)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# 2. Node.jsをインストール
brew install node

# 3. pnpmをインストール
brew install pnpm

# 4. 新しいターミナルを開き直す、または以下でPATHを反映
source ~/.zprofile

# 5. 確認
node --version
pnpm --version
```

> Node.js/pnpmはHomebrew以外(nvm、公式インストーラ等)で入れても構いません。
> `node --version` / `pnpm --version` が通ればOKです。

### `command not found: pnpm` が出た場合の対処

1. **まず新しいターミナルウィンドウ/タブを開き直してみてください。**
   Homebrewインストール直後の既存ターミナルは、古いPATH設定のままのことがあります
2. それでも直らない場合:
   ```bash
   source ~/.zprofile
   ```
3. `brew` 自体が無い場合は、上記「インストール」の手順1から実施してください
4. Node.jsは入っているのに `pnpm` だけ無い場合(Node.js 25以降は`corepack`が同梱されないことがあります):
   ```bash
   corepack enable && corepack prepare pnpm@latest --activate
   # または
   brew install pnpm
   ```

## 処理の流れ

検索条件に一致しただけでは応募対象にしません。「案件を探す」と「応募すべきか判定する」を
分離しています。

```
① search-conditions.yaml の条件で案件を検索(粗い絞り込み)
    ↓
② 案件詳細(本文)を取得
    ↓
③ job-criteria.yaml の基準をもとに、Claudeが本文まで読んで採用/除外を判定
    ↓
④ 採用された案件だけ、profile.yaml の情報から応募文ドラフトを生成
```

## できること

1. 実際にクラウドワークスで検索・絞り込みを行った検索結果ページのURLをもとに、新着案件を取得
2. すでに見た案件は除外（重複通知なし）
3. 契約方式（固定報酬 / 時間単価）ごとに適切な金額条件でフィルタ（あくまで目安、詳細は下記）
4. 除外キーワード・新着のみ・募集中のみ、といった条件でも絞り込み可能
5. 検索条件に一致した案件でも、本文まで読んだ上でAIが応募対象として妥当かを判定
6. 応募対象と判定された案件だけ、あなたのプロフィールをもとにAIが応募文ドラフトを作成
7. `data/reports/` にMarkdownレポートを出力（採用案件のドラフトと、除外した案件・理由の一覧）

## ログイン・セッションの仕組み

- 初回実行時は、クラウドワークスへのログイン(ID/パスワード)を自動で行います
- **2段階認証(メールで届く6桁コード)が設定されている場合**、ヘッドフルモード
  (`pnpm run run:headful` / `pnpm run run:test`)で実行すると、認証コード入力待ちで
  一時停止します。開いているブラウザでコードを入力したら、ターミナルに戻って
  **Enterキーを押してください**(詳しい実行例は下記「初回ログイン(2段階認証)の流れ」参照)
- ログインに成功すると、セッション情報(Cookie等)を `data/session.json` に保存します
  - このファイルは `.gitignore` で除外されており、**Gitには一切含まれません**
  - 次回以降の実行では、まずこの保存済みセッションを再利用し、有効であれば
    **認証コードの入力なしに**そのまま処理を続けます
  - セッションが失効していた場合のみ、自動的に再ログイン(必要なら2段階認証の
    一時停止)が発生します
- ヘッドレスモード(`pnpm run`)でセッションが無い/失効しており、かつ2段階認証が
  必要な場合はエラーで停止します。その場合は一度 `pnpm run run:headful` を実行して
  セッションを作成してください

### 初回ログイン(2段階認証)の流れ

```bash
pnpm run run:test
```

1. ヘッドフルモードでブラウザが開き、自動でID/パスワードを入力してログインを試みます
2. 2段階認証画面が出たら、ターミナルに以下のように表示されます
   ```
   ========================================
   2段階認証が必要です。
   開いているブラウザ画面で、メールに届いた認証コードを入力し「認証する」を押してください。
   完了したら、このターミナルに戻って Enter キーを押してください。
   認証コード入力後、Enterを押してください:
   ```
3. 登録メールアドレスに届いた6桁のコード(有効期限10分)を、**開いているブラウザ画面上で**入力し「認証する」を押す
4. ブラウザでの認証が完了したら、ターミナルに戻って **Enterキー**を押す
5. ログイン状態が `data/session.json` に保存され、以降の処理が続行されます

## セットアップ(初回)

**通常のmacOSターミナル**で以下を実行してください(Cursor/Claude Code等のツールは不要です)。

```bash
cd crowdworks-automation
pnpm install
cp .env.example .env
pnpm run setup
```

- `.env` にクラウドワークスのログイン情報と Anthropic API キーを入力してください
- `pnpm run setup` は環境全体を診断するコマンドです。以下を確認し、問題があれば
  具体的な対応コマンドを表示します(詳細は下記「環境セットアップの確認(pnpm run setup)」参照)
  - Node.jsが導入されているか・バージョンは十分か
  - pnpmが使用できるか
  - Playwright Chromiumがインストールされているか
  - `.env` が存在し、必要な項目が入力されているか
  - 設定ファイル(`*.yaml`)が存在するか
- 設定ファイルがまだ無い場合は、続けて以下を実行してください
  ```bash
  pnpm run config:update
  ```
  `*.example.yaml` をもとに `profile.yaml` / `search-conditions.yaml` / `job-criteria.yaml` が
  自動作成されます(詳細は下記「設定ファイルの管理の仕組み」参照)
- その後、以下を編集してください
  1. `search-conditions.yaml` : 検索条件（`searchUrl` を実際の検索結果URLに差し替え。詳細は後述）
  2. `job-criteria.yaml` : 応募対象の判定基準（詳細は下記「応募対象判定」参照）
  3. `profile.yaml` : 名前・強み・ポートフォリオ・稼働条件など（詳細は下記「応募者プロフィール」参照）
- 準備ができたら、まずは安全なテストモードで動作確認してください
  ```bash
  pnpm run run:test
  ```

## 環境セットアップの確認(pnpm run setup)

```bash
pnpm run setup
```

Node.js/pnpm/Playwright/設定ファイルの状態をまとめて確認する診断コマンドです。
`config.ts`(本体のロジック)には依存しない、独立したチェック専用のスクリプト
([src/cliSetup.ts](src/cliSetup.ts))なので、`.env` が無い状態でも安全に実行できます。

- ✅ 問題なし / ⚠ 動作はするが要確認 / ❌ 対応が必要、の3段階で表示します
- ❌ の項目には、実行すべき具体的なコマンドが添えられます
- 何か問題が起きたときの切り分けにも使えます(まずこのコマンドを実行してみてください)

## 設定ファイルの管理の仕組み

このツールの設定ファイルは、すべて **「`○○.example.yaml`(テンプレート) + `○○.yaml`(あなたの実際の設定)」**
のペアで管理されています。ファイル名を個別にハードコードした仕組みではなく、
**`*.example.yaml` を置くだけで自動的に管理対象として認識される**汎用の仕組みです
([src/configSync.ts](src/configSync.ts))。新しい設定ファイルを増やしたくなったら、
コードは一切変更せず `foo.example.yaml` を置くだけで、`config:check` / `config:update` の対象になります。

### ルール

- **実際の `.yaml` が存在する場合は絶対に上書きしません。** あなたが入力した値・URL・APIキー・
  応募文などは、明示的に削除しない限り保持されます
- 実際の `.yaml` が**存在しない場合だけ**、対応する `.example.yaml` から自動生成されます
- `.example.yaml` 側に新しい項目が追加された場合(例: バージョンアップで `newOption:` が追加された等)、
  実際の `.yaml` 側には**不足している項目だけ**を追記します。既存の値・配列の中身は一切変更しません
- 更新前には必ずバックアップ(`○○.yaml.backup`)を作成します
- `pnpm run config:update` は、実際に書き込む前に**追加・変更予定の内容を一覧表示し、
  「続行しますか？ (Y/N)」の確認を必ず挟みます**。`N`(またはYes以外の入力)ならファイルは
  一切変更されません。誤って実行してしまっても取り消せます
- 実行結果には「追加された項目」「保持した項目」「手動確認が必要な項目」を表示します
- 設定ファイルの中身(APIキーなど機密情報を含む可能性がある値)はログに出力しません。表示されるのは
  項目名(キーのパス)のみです
- `pnpm run config:check` は差分の有無にかかわらず正常終了(終了コード0)します。差分があってもエラー扱いにはしません

### コマンド

```bash
# example との差分を確認するだけ(何も書き換えません)
pnpm run config:check

# バックアップを作成した上で、不足項目だけを安全に追記します
pnpm run config:update
```

`pnpm run` / `pnpm run:headful` の実行時にも、起動時に自動で差分チェックが走ります。
ただし自動で書き換えるのは「ファイルがまだ存在しない場合の新規作成」だけで、
既存ファイルへの追記は行いません(不足があれば警告を表示するだけです)。
実際に追記したいときは `pnpm run config:update` を明示的に実行してください。

### 新しい設定ファイルを追加する方法

1. `crowdworks-automation/` 直下に `foo.example.yaml` を作成し、テンプレートとなる項目を書く
2. `pnpm run config:check` を実行すると、`foo.yaml` が未作成であることが表示されます
3. `pnpm run config:update` を実行すると、`foo.yaml` が自動作成されます
4. 以降、`foo.example.yaml` に項目を追加したら、`pnpm run config:update` で
   `foo.yaml` 側にも不足分だけが安全に追記されます

コード側(`src/config.ts` など)でこの新しい設定ファイルを実際に読み込んで使うには、
別途 `loadXxx()` のような読み込み関数を追加する必要がありますが、
**ファイルの生成・同期・バックアップの仕組み自体はコード変更不要**です。

## 既知の制約・要確認事項

- `src/scraper.ts` 内のCSSセレクタはCrowdWorksの現行UIをもとにした推測値です。
  実際のログイン・案件一覧取得で動作しない場合は、`pnpm run:headful` で
  ブラウザの挙動を見ながら調整が必要です。
- 金額フィルタ（`minBudget` / `minHourlyRate`）・`newOnly`・`openOnly` は、
  いずれも一覧カードのテキストから該当しそうな数値・表示を推定しているだけなので、
  誤って除外/通過するケースがあります。最終判断はレポートを見て自分で行ってください。
  （判定ロジックは [src/filters.ts](src/filters.ts) にまとまっています）
- クラウドワークス利用規約 第23条12号により、サーバーに過剰な負荷をかける行為は禁止されています。
  検索条件（`searchUrl`）の数を増やすほど1回の実行で発生するアクセス数も増えるため、
  条件数が多い場合は実行頻度を1日1〜2回程度に抑えるなど、バランスを取ってください。
- 応募対象判定(`job-criteria.yaml`)はAIによる判断のため、誤って採用/除外するケースがゼロではありません。
  除外した案件も理由付きでレポートに残しているので、定期的に見直して基準を調整してください。
- 検索条件に一致した案件ごとに「応募対象判定」と「(採用時のみ)応募文生成」でAPI呼び出しが発生します。
  条件数・案件数が多いとAnthropic APIの利用量が増える点に留意してください。

## 応募対象判定（job-criteria.yaml）

`search-conditions.yaml` に一致した案件は、まだ「候補」でしかありません。
実際に応募対象とするかどうかは、[src/screening.ts](src/screening.ts) が
`job-criteria.yaml` の基準をもとに、**案件の本文まで読んだ上で**Claudeに判定させます。

- `acceptKeywords` / `rejectKeywords` は参考情報として渡すだけで、**機械的なキーワード一致では判定しません**
  - タイトルに歓迎キーワードがなくても、本文の内容が合致していれば採用
  - タイトルに歓迎キーワードがあっても、本文の内容が除外対象なら不採用
- `description` に自由記述で判定方針を書くと、キーワードだけでは表現しづらいニュアンスも反映されます
- 判定結果は `data/reports/` のレポートに、採用理由・除外理由つきで出力されます
  - 誤って除外されていそうな案件があれば、`job-criteria.yaml` のキーワードや `description` を調整してください
- コードの変更は不要、このファイルを編集するだけで次回実行から反映されます

| 項目 | 必須 | 内容 |
|---|---|---|
| `acceptKeywords` | ○(空リスト可) | 歓迎したい案件の特徴(参考情報) |
| `rejectKeywords` | ○(空リスト可) | 避けたい案件の特徴(参考情報) |
| `description` | - | 判定方針の自由記述の補足 |

## 応募者プロフィール（profile.yaml）

応募文生成に使われる情報は **`profile.yaml` の1ファイルだけ**です（[src/config.ts](src/config.ts) の
`loadProfile()` が読み込み、[src/draftGenerator.ts](src/draftGenerator.ts) がAIへの指示文に埋め込みます）。

- コードを触らずに、このファイルを編集するだけで**以後生成されるすべての応募文に即反映**されます
- `name`（名前）・`strengths`（強み、1件以上）は必須項目です。足りないと実行時にエラーで教えてくれます
- 名前・肩書き・稼働条件などが変わったら、その都度このファイルだけ書き換えればOKです

| 項目 | 必須 | 内容 |
|---|---|---|
| `name` | ○ | 応募文の署名に使う名前 |
| `strengths` | ○ | 強み・実績のリスト |
| `title` | - | 肩書き・キャッチコピー |
| `portfolio.url` / `portfolio.note` | - | ポートフォリオURLと説明 |
| `availability.hours` / `availability.scope` | - | 稼働時間・対応範囲 |
| `extraNotes` | - | その他アピールしたいこと(リスト) |
| `sampleApplication` | - | 良いと思った応募文サンプル(文体の参考用) |

## 検索条件（search-conditions.yaml）の増やし方

`searches:` の下にブロックを追加するだけです。同じジャンルでも契約方式ごとに
ブロックを分けて登録できます（例: 「Python開発(固定報酬)」「Python開発(時間単価)」）。

```yaml
  # 固定報酬の場合
  - name: "新しいジャンル名(固定報酬)"
    searchUrl: "（クラウドワークスで実際に検索・絞り込みを行った結果ページのURL）"
    contractType: fixed        # 省略可(既定でfixed)
    minBudget: 30000           # 最低契約金額(円)
    excludeKeywords: ["除外したい語"]
    newOnly: false
    openOnly: true

  # 時間単価の場合
  - name: "新しいジャンル名(時間単価)"
    searchUrl: "（クラウドワークスで実際に検索・絞り込みを行った結果ページのURL）"
    contractType: hourly
    minHourlyRate: 2000        # 最低時給(円)
    openOnly: true
```

| 項目 | 必須 | 内容 |
|---|---|---|
| `name` | ○ | 自由な管理用ラベル（レポートに表示されます） |
| `searchUrl` | ○ | クラウドワークスで実際に検索・絞り込みを行った結果ページのURL |
| `contractType` | - | `fixed`(固定報酬・既定) または `hourly`(時間単価) |
| `minBudget` | - | `contractType: fixed` のときの最低契約金額(円) |
| `minHourlyRate` | - | `contractType: hourly` のときの最低時給(円) |
| `excludeKeywords` | - | タイトル・案件情報にこの語を含む案件を除外(リスト) |
| `newOnly` | - | `true` で「新着」表示がある案件のみ対象 |
| `openOnly` | - | `true` で募集終了・受付終了の案件を除外 |

- `searchUrl` は **必ず、クラウドワークスにログインした状態で実際に検索・絞り込みを行い、
  表示された検索結果ページのURLをそのままコピーしたもの**を設定してください
  （`keyword=` だけの簡易URLは条件の再現性が低いため使用しません）
- `minBudget` と `minHourlyRate` はどちらか片方だけ設定してください
  （`contractType: hourly` なのに `minBudget` を書く、といった組み合わせはエラーになります）
- YAMLの構文（インデント・コロンの後の半角スペース）が崩れるとエラーになるので、
  既存のブロックをコピーして書き換えるのが安全です
- 起動時（`pnpm run` / `pnpm run:headful`）に形式チェックが入るので、
  記述ミスがあれば実行前にエラーメッセージで教えてくれます
